# Agent 学习（二）——Environment 与 Agent 的 gap

> 上一节我们得出了一个结论：LLM 只能产生信息，不能改变世界。这一节我们把这个结论放进架构里来看。

## 一个非常重要的概念：Environment（环境）

先别害怕，它其实很好理解。

以前我们一直说"环境在变化"。那什么叫环境？

**定义：**

> Agent 能够观察和影响的一切东西，就叫 Environment（环境）。

例如：

**如果 Agent 操作 Windows：**

```
Windows
├── 文件
├── 鼠标
├── 键盘
├── 浏览器
├── 微信
├── 桌面
└── 应用程序
```

**如果 Agent 操作浏览器：**

```
浏览器
├── 网页
├── 输入框
├── 按钮
├── 图片
└── 链接
```

**如果 Agent 控制机器人：**

```
现实世界
├── 桌子
├── 杯子
├── 门
├── 人
└── 障碍物
```

所以，Agent 从来不是活在真空中的。**它一定有一个环境。**

## 当前架构

到目前为止，我们只有两个角色：

```
          User
            │
            ▼
        +--------+
        |  LLM   |
        +--------+
            │
     生成文字（信息）
            │
            ▼
      用户自己执行
```

**问题出在哪？**

LLM 和环境之间，**没有任何连接。**

它知道世界，但是**碰不到世界。**

## Failure Case：如果只有 LLM，会发生什么？

假设只有 LLM，没有任何其他能力。

**它可以：**

- ✅ 理解需求
- ✅ 分析问题
- ✅ 提供建议
- ✅ 制定方案

**但是它不能：**

- ❌ 打开软件
- ❌ 删除文件
- ❌ 浏览网页
- ❌ 查询实时数据
- ❌ 点击按钮
- ❌ 改变环境

所以，它离真正的 Agent，还差一步。

## 那下一步应该增加什么？

这是今天课程最后的问题，也是下一课的内容。

现在架构是：

```
User
  │
  ▼
 LLM
```

我们希望变成：

```
User
  │
  ▼
 LLM
  │
 ????
  │
Windows
```

**这里的 ???? 到底是什么？**

很多教程会直接告诉你：Tool。

但我不会这么讲。因为我希望你自己把这个组件"发明"出来。

## 今天的思考题

还是三个问题，不过这次稍微深入一点。

**问题 1：**

为什么我说"LLM 最大的缺陷不是'不会调用工具'，而是'不能改变环境'"？

请试着用自己的话解释，不要直接重复原文。

**问题 2：**

如果给 LLM 一个 API：`open_wechat()`，它是不是立刻就变成 Agent 了？为什么？

（这个问题没有标准答案，更想看你的思考过程。）

**问题 3：**

请你站在"系统设计者"的角度想一想。

假设你正在设计第一代 Agent。现在只有：

```
User
  │
  ▼
 LLM
```

为了让它能够真正"行动"，你觉得应该在 LLM 和 Windows 之间增加一个什么样的组件？

不要说名字（比如 Tool）。试着描述它应该承担什么职责。

---

最后说一句我自己的观察。

如果你在跟这个系列，你的思维方式可能很适合这种学习路线——你回答问题时，关注的是**为什么**，而不是急着记术语。这正是做架构设计最重要的习惯。

从这一课开始，后面会慢慢加入一些工程视角。你会发现，我们不是在学一个 AI 新名词，而是在设计一个**软件系统**。而如果你有 Java 后端和游戏框架的经验，会越来越派上用场。
