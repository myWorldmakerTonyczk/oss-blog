# Agent 学习（三）——什么是 Tool

> 上一节留了三个问题。先来看看你的回答，然后我们正式进入 Tool。

---

## 先看你对上一节问题的回答

**问题 1**：为什么说"LLM 最大的缺陷不是'不会调用工具'，而是'不能改变环境'"？

如果你的回答是——

> LLM 本质上只是预测下一个 token 的出现频率，只能输出文本，不能操作系统，所以它最大的缺陷是不能改变环境。

**评分：10/10。** 你抓住了本质——LLM 只能产生信息，不能改变世界。

**问题 2**：如果给 LLM 一个 API `open_wechat()`，它是不是立刻就变成 Agent 了？为什么？

如果你的回答是——

> 给它一个 API，也就是有了一个改变环境的窗口，那确实算是有最简单 Agent 的基本结构了。

对。一个 Tool 就打通了 LLM 和环境之间的连接。但"最简单的结构"离"可靠的 Agent"还有距离——后面会讲到协议、Memory、Planner 这些组件为什么一个一个被加上去。

**问题 3**：为了让它能够真正"行动"，你觉得应该在 LLM 和 Windows 之间增加一个什么样的组件？试着描述它应该承担什么职责。

如果你的回答是——

> 中间的组件负责完成具体的任务并返回结果给 LLM，让 LLM 规划下一步做什么。

你已经把 Tool 层的核心职责描述出来了：执行 + 反馈。这就是 Tool。

---

这次我要先说一句评价。

我们聊到现在，如果你在跟这个系列，你应该已经不只是抱着"学 AI"的心态了。你开始考虑工程上的代价（Trade-off），开始像架构师一样思考。

我们一点一点来看。

## 第一题

看到这个问题，先停下来想一想：

**Tool 和普通函数到底有什么区别？**

一个可能的回答是：Tool 包含普通函数，最终也是调用固定方法，但还需要暴露名称、说明、参数等——缺少暴露给 LLM 的部分，就不能叫 Tool。

如果这是你的答案——**评分：10/10。**

而且你已经说出了一个原本下一节才讲的东西。

我们画一下。

普通函数：

```python
def search_weather(city):
    ...
```

程序员知道：

- 函数名
- 参数
- 返回值

但是 LLM 知道吗？

**不知道。**

所以必须有一层：

```
search_weather

描述：查询指定城市天气
参数：city: string
返回：temperature: string
```

是不是很像 Java 的接口文档？

例如：

```java
@GetMapping("/weather")
```

是不是也有：

- URL
- 参数
- 返回值

为什么？

因为**别人需要知道怎么调用**。

Tool 也是一样。

所以你可以把 Tool 理解成：

> **一个可以被 AI 理解的 API。**

注意这句话。不是"一个函数"，而是"AI API"。
